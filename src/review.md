## 2차 상속 추가 고민 사항

### 1. CLI 흐름

해당 프로그램은 의도적으로 재고 없음, 불가능한 옵션 선택의 가능성을 열어두고 재주문을 유도합니다.

실제 키오스크는 품절 메뉴는 아예 선택하지 못하게 하여 에러 가능성을 최소화하지만,
이번 프로젝트는 과제이기 때문에 order() 안에서 다른 메서드를 호출하는 것과 잘못된 선택을 하면 재주문할 수 있도록 연습하기 위해 이 방식으로 기획하였습니다.

### 2. count, price 필드 위치

- Product 클래스를 추가하고 위 두 필드를 Product, Drink 중 어디에 두어야 할지 고민이 되었다.
- 직원 호출과 같은 서비스가 추가된다면 두 필드가 필요하지 않기 때문에 음료 전용 속성으로 Drink에 두어야 할까? 생각하였다.
- 하지만, 다른 Product 하위 클래스에도 두 필드가 필요하고 Service와 Product는 상속 관계가 아니라는 결론을 내려 Product에 두 필드를 두었습니다.
    - Service is Product 성립 X

### 3. 필드 isIce → isHot → tempOption 변경

초기 설계(사진)에서는 아이스 여부를 `isIce` 필드로 관리하려 했습니다.

하지만 코드를 작성하다보니, 에이드는 항상 아이스여야 하는데, isIce 필드와 중복되었습니다.

이를 해결하기 위해 음료를 기본적으로 아이스로 하여 뜨거운 음료가 가능한 음료를 isHot으로 선택할 수 있도록 하는 방향으로 수정하였습니다.

이 수정 사항에도 걸리는 점이 있었습니다.

- isHot 필드가 가능 여부인지, 선택인지 헷갈릴 수 있음
- 따뜻한 음료만 되는 경우가 없음

이런 이유로 앞의 고민에서 언급했던 번호로 온도를 선택하는 방법으로 수정하여 ice only, hot only, both 선택이 가능하도록 수정하였습니다.

### 4. 클래스 Juice → NonCoffee 변경

위 고민에 이어 기존 Juice 클래스에 뜨거운 음료가 필요 없을 것 같다는 생각이 들었습니다.

- (따뜻한 수박 주스, 따뜻한 오렌지 주스 등이 어색함)

이에 생각한 방향이 차(tea) 종류였고, 주스로 한정짓지 않고 논커피 음료를 포함하는 방식으로 수정하였습니다.

### 5. Product order() 추상메서드?

초반에는 order()메서드를 Coffee, NonCoffee 클래스의 order()에서 반드시 구현하도록 강제하고, Product에서는 주문이 불가하도록 하기 위해 추상메서드를 생각하였습니다.

하지만, 추상메서드는 시그니처가 고정되어야 하기 때문에 매개변수가 같아야 해서 해당 프로그램의 구조와 맞지 않았습니다.

때문에 아래 세 가지 대안을 고민하였습니다.

1. setter을 사용해 매개변수를 미리 필드에 세팅한 후 order() 호출
    - setter → order() 호출 순서를 강제할 수 없기 때문에 setter 없이 order()를 호출하거나 이전 주문의 옵션이 필드에 남아있게 되면 상태 불일치가 발생할 수 있기 때문에 제외
2. 새로운 객체를 만들어 필요한 옵션을 묶어 order~~ 객체 형태로 통일
    - 위의 문제는 해결되지만, CoffeeOrderRequest, NonCoffeeOrderRequest 같은 클래스가 추가되어 계층이 늘어나고 오히려 복잡해진다고 판단하여 제외
3. 오버로딩
    - Coffee와 NonCoffee의 order()에 필요한 각 클래스의 옵션을 메서드에 표현할 수 있고,
      Drink의 order()를 super.order()로 호출하여 재고 확인, 음료 온도 검증 같은 공통 로직을 재사용하면서 하위 클래스 별 추가 검증만 구현할 수 있기 때문에 자연스럽다고 생각해 선택

+ Product는 공통 속성을 가지는 상위 개념이고 실제 판매 가능한 객체는 하위 클래스에만 존재하기 때문에 abstract로 선언하였습니다.

### 6. Main.java 작성 방식

계획한 CLI 흐름이 분기되고 주문 불가능한 옵션 선택 시 반복도 해야하기 때문에 모든 메서드를 메인 클래스에 넣기보다 인스턴스를 분리하는 방식을 선택하였습니다.

다른 방법으로 static 전역 필드, 메서드를 사용하는 방식은 구현이 간단해진다는 장점이 있지만, static 필드는 프로그램 전체에서 공유되기 때문에 동일한 CafeApp에서 여러 흐름에서 사용하거나 이전 상태가 남아 예상하지 못한 동작이 발생할 수 있습니다.

파라미터로만 전달하는 방식은 여러 메서드에서 Scanner를 반복적으로 전달해야 하고, 메서드 호출도 반복돼 필요 이상으로 코드가 길어질 수 있습니다.

진행한 방식은 Scanner를 생성자로 주입받기 때문에 Main.java는 Scanner 생성과 실행만 담당하고 CafeApp은 주문 흐름 전체를 담당하도록 책임을 분리할 수 있습니다.
